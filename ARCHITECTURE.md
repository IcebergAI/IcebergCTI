# Iceberg — Architecture & subsystem reference

> Companion to [CLAUDE.md](CLAUDE.md); the detailed reference deliberately kept out of the always-on agent context.

## Architecture (detail)

_The lean CLAUDE.md keeps a compressed auth summary and a one-line-per-invariant list; the full paragraphs live here._

Authentication is **multi-provider OIDC** (Microsoft Entra, Authentik, Auth0, Okta — simultaneously; Authorization Code flow with PKCE), admin-configured on the `OIDCSettings` singleton row (`/admin/oidc`, seeded from the legacy `ICEBERG_OIDC_*` env on first read). One generic Authlib registration per **enabled** provider (`services/oidc_settings.enabled_providers`) plus a per-IdP **adapter** (`auth/oidc/`: a `StandardOIDCAdapter` base + thin `entra`/`authentik`/`auth0`/`okta` claim-extraction subclasses in a self-registering registry) drives a single parametrised route pair `/auth/oidc/{provider}/{login,callback}`; the legacy `/auth/entra/login` + `/auth/callback` remain Entra back-compat aliases. A per-provider `role_map` (`"group=ROLE,…"`) maps the IdP's group/role claim to an Iceberg role (unmapped/absent → read-only `STAKEHOLDER`; Entra's >200-group "overage" fails closed); a missing **or explicitly unverified** email is rejected, and provisioning is keyed on the immutable `(auth_provider, issuer, sub)` triple (email is non-identifying — the same person may exist under two IdPs — and a cross-provider `(issuer, sub)` collision is refused). Each provider's client secret is **env-only** (`ICEBERG_OIDC_<PROVIDER>_CLIENT_SECRET`). Optional Entra profile claims (`department`, `jobTitle`, `companyName`, `officeLocation` by default) are persisted on the user for stakeholder categorisation. After login Iceberg mints its own short-lived JWT with a per-user `token_version` claim — sent as a Bearer header by API clients, or stored in a signed session cookie by the portal — so logout can invalidate existing Iceberg tokens and the "all endpoints JWT-authenticated" rule holds uniformly. A **dev-login bypass** (`ICEBERG_DEV_AUTH=true`, disabled when `ICEBERG_ENVIRONMENT=prod`) issues a JWT for a chosen role without an IdP, for local development and tests. The JWT and the OIDC session cookie are signed with **purpose-separated keys** derived from `ICEBERG_SECRET_KEY` via HMAC-SHA256 (`auth/signing.py`, `jwt` vs `session` contexts), so a token from one signing context can never validate in the other. A prod instance with no usable login path (dev auth hard-off, OIDC unset) logs a startup warning rather than failing boot (`main._warn_if_no_login_path`).

The session cookie is `SameSite=Lax` + `HttpOnly` (and `Secure` in prod); as defence-in-depth a **same-origin CSRF middleware** (`auth/csrf.py`) rejects any cookie-authenticated state-changing request whose `Origin`/`Referer` doesn't match the host (Bearer API clients and anonymous requests are exempt — token auth isn't browser-CSRF-prone). Logout is **POST-only** so it can't be triggered cross-site.

**Rate limiting (`auth/rate_limit.py`).** A token-bucket `RateLimitMiddleware` protects the abuse-prone surfaces with per-policy budgets (`build_policies`): the auth endpoints (dev-login POSTs and the OIDC login/callback, keyed **per-IP** per minute) and the expensive/outbound actions — AI assist (`/api/ai/*` except `accept-provenance`), report PDF renders, the admin connectivity tests + MISP pushes, and search — keyed **per-user** (falling back to IP for anonymous requests) per hour/minute with a configurable burst. It is **on by default in prod only** (`ICEBERG_RATE_LIMIT_ENABLED` unset → `settings.rate_limit_active` keys off `is_prod`); the backing store is **in-memory** for a single process or **Redis** for shared cross-worker buckets (`ICEBERG_RATE_LIMIT_STORE=auto|redis|memory` + `ICEBERG_RATE_LIMIT_REDIS_URL`; `redis` without a URL refuses to boot, prod-on-memory logs a warning). A store outage **fails open by default** (`ICEBERG_RATE_LIMIT_FAIL_OPEN`, else 503). Throttled requests get a 429 (JSON for `/api/*`, plain text for the portal) with `Retry-After` + `X-RateLimit-*` headers, and are **audited** (`AuditAction.RATE_LIMITED`, WARNING) with the SIEM emit scheduled off the response path. Middleware order (outermost → innermost) is `SecurityHeaders → Audit → RateLimit → Session → CSRF`, so 429s are audited and still carry security headers. Per-policy budgets are tuned by the `ICEBERG_RATE_LIMIT_*` env vars (all validated ≥ 1).

**Security response headers (`auth/security_headers.py`).** A `SecurityHeadersMiddleware` (registered **outermost** in `main.py`, after `AuditMiddleware`, so it stamps every response including middleware-level 401/403s) sets the standard set on every response: a **strict Content-Security-Policy** (`default-src 'self'`; `script-src 'self'` with **no `'unsafe-inline'`/`'unsafe-eval'`**; `img-src 'self' data:` for the base64 figure embeds; `object-src 'none'`; `base-uri`/`form-action 'self'`; `frame-ancestors 'none'`; `upgrade-insecure-requests` in prod), plus `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy`, `Permissions-Policy` (deny camera/mic/geo/usb/…), `Cross-Origin-Opener-Policy`, and **HSTS only in prod** (`is_prod`-gated — the app behind a TLS-terminating proxy can't key off the request scheme). `style-src` keeps `'unsafe-inline'` deliberately (dynamic inline `style=` attributes can't be nonced; CSS injection is far lower-risk than script). The strict `script-src` is what makes the portal carry **no inline JavaScript** — see *Self-hosted assets + SRI* for the Alpine CSP build. The header values are a pure function `build_security_headers(settings)` (unit-tested dev vs prod); headers are applied with `setdefault` so a per-route header (e.g. the file-download `nosniff`) is never duplicated.

**Health probes (`health.py`).** Two **unauthenticated** root-level endpoints (registered before the auth/api/web routers, no `/api` prefix, no auth dependency, kept **out of the OpenAPI schema**) back container orchestration: `GET /healthz` (**liveness** — process up, no DB touch, always `200 {"status":"ok"}`; a transient DB blip must not restart the pod) and `GET /readyz` (**readiness** — queries a core table via the shared session, so it reflects *schema* readiness, not just connectivity: `200 {"status":"ready"}` or `503 {"status":"not ready"}`). Readiness queries `user` rather than a bare `SELECT 1` because the prod deploy runs migrations separately (`ICEBERG_AUTO_MIGRATE=false`) — "process up" ≠ "schema ready". GET/200/503 sail past the middleware stack (CSRF exempts GET, `AuditMiddleware` only records 401/403, the auth-redirect handler only fires on 401).

## Domain model — full detail

- **User** — identity keyed on the immutable `(auth_provider, issuer, sub)` triple (email is **not** an identity key and is no longer globally unique — a person may exist under two IdPs), role, optional `preferred_intel_level`, per-user `token_version` for logout/session revocation, optional OIDC organisation attributes, tag subscriptions and audience-group memberships.
- **Notebook** — topic workspace owned by an analyst; has many sources, notes and reports. Collection material (the notebook, its sources/notes/attachments/diamonds) is **writer-only**: read-only stakeholders never list or open notebooks — they consume only finished products (reports/feed/search). Create/read helpers live in `services/notebooks.py` and are shared by the API and portal. **Access is role-wide, not owner-scoped (#65)** — notebooks are a *shared team workspace*: any writer (ANALYST/REVIEWER/ADMIN) may open and edit any notebook and its material (sources/notes/attachments/figures/diamonds/ACH/IOCs), so multiple analysts can collaborate on one topic/request. `Notebook.owner_id` is **provenance/attribution only (who created it), not a general access-control boundary** — content-editing writer routes gate on `_require_writer` (role), never on ownership. **The one deliberate exception is deleting the whole notebook**, restricted to its **owner (or an admin)** (`api/notebooks.delete_notebook` checks `owner_id`): dropping a notebook destroys all its collected material at once, so it's treated as more destructive than editing within it and kept a natural owner-only restriction — collaboration happens *inside* a notebook, not by deleting a peer's. (Distinct from compartmentation of *published products*, #33.)
- **Source** / **Note** — collected material inside a notebook. Sources carry a
  **`tlp`** handling marking (defaults to **AMBER** for manual entry, stamped
  **CLEAR** for RSS-ingested articles in `feeds.send_item_to_notebook`) which
  **gates AI egress of the source's content** (`ai.should_send_source` against
  `ICEBERG_AI_MAX_TLP`, applied to `summarise-source` + `ioc_extract`) and is
  **inherited by IOCs** captured from the source. Sources also carry
  Admiralty/NATO-style grading (`reliability` A-F + `credibility` 1-6), with
  provenance (`UNGRADED` / `AUTO` / `MANUAL`) and a rationale.
  `services/source_grading.py` owns the **offline heuristic grader** (reliability
  inferred from source identity, credibility from the analyst summary plus pasted
  source content in `content_md`) plus manual override / clear / regrade. There is
  no server-side remote URL fetch or LLM call in the grading path.
- **Attachment** — an uploaded reference file held against a notebook. Stored on disk under `ICEBERG_ATTACHMENTS_DIR` with a server-generated UUID name; the DB row keeps metadata + the original filename. Upload/download are **writer-only** (read-only stakeholders have no access); uploads are MIME-whitelisted (env-tunable via `ICEBERG_ATTACHMENT_ALLOWED_TYPES`; SVG deliberately excluded), size-capped (`ICEBERG_ATTACHMENT_MAX_MB`, default 25), and **byte-validated** after write (`services/upload_validation.py` sniffs magic bytes / container structure per declared MIME type — PNG/JPEG/GIF/WebP signatures, `%PDF-`, OLE and OOXML structure, NUL-free text — so a renamed executable is rejected regardless of its claimed type). Citable in reports via `ReportAttachment` and listed in the rendered PDF's appendix. See `services/attachments.py`.
- **DiamondModel** — a Diamond Model of Intrusion Analysis assessment held against a notebook (the four core features `adversary`/`capability`/`infrastructure`/`victim` + an analytic `confidence` + notes). Rendered server-side to an **SVG** diagram and **embedded inline in a report** by writing a `[[diamond:ID]]` token in the markdown body — there is **no citation link table**; the token is the association, resolved (notebook-scoped) at render time for the web view, the live preview, and the Typst PDF (Typst renders SVG natively; vertex text is XML-escaped). See `services/diamond.py`.
- **Figure** — an uploaded image (PNG/JPEG/GIF) held against a notebook and **embedded inline in a report** by writing a `[[figure:ID]]` token in the markdown body. Like DiamondModel there is **no citation link table** — the token is the association, resolved (notebook-scoped) at render time. Stored on disk under `ICEBERG_FIGURES_DIR` with a server-generated UUID name (capped by `ICEBERG_FIGURE_MAX_MB`, default 10, and byte-validated by the same `services/upload_validation.py` magic-byte pass as attachments); the figures **collection** is writer-only. Inlined as a base64 `data:` URI in the web view / live preview (injected after nh3 sanitisation, like diamond SVGs) and copied into the Typst `--root` for the PDF. See `services/figures.py`.
- **IOC** — an indicator of compromise captured against a notebook (light-touch IOC FR). A **transient staging** entity, not an IOC database — the authoritative store is external (**MISP**). Carries an `ioc_type` (`IOCType`, whose **enum value is the MISP attribute type string** so the push maps 1:1 — `ip-src`/`ip-dst`/`domain`/`hostname`/`url`/`md5`/`sha1`/`sha256`/`email-src`/`filename`/`vulnerability`), a `value`, free-text `description`, a **`tlp`** handling marking (**inherited from the provenance source** when one is set — an explicit value wins, else AMBER — and stamped per-attribute on the MISP push), and an optional `source_id` (provenance, `SET NULL`, scoped to the same notebook). Entered **manually**, or **AI-suggested** from a source's text (the governed `ioc_extract` task — FR #95 — refanged + `IOCType`-constrained by `iocs.normalise_candidates`, then promoted one-by-one through the ordinary create endpoint; see *Governed AI assist*). A report **cites a subset** via the **`ReportIOC`** link table (`Report.cited_iocs`, like `ReportSource`) — rendered as a structured **Indicators appendix** in the web view + Typst PDF and pushed to MISP. Writer-only collection. See `services/iocs.py` and *Light-touch IOCs (MISP push)*.
- **ReportMispEvent** — records a report's push to MISP (light-touch IOC FR): one row per report (unique `report_id`) holding the external `event_uuid`/`event_id`, the `attribute_count`, and the last outcome (`last_status`/`error`/`pushed_at`). Enables **idempotent re-push** (update the same MISP event) and surfaces success/failure on the report view. Modelled on `DisseminationEvent`; kept off the immutable `Report` row.
- **MISPSettings** — the single-row (id=1) **outbound MISP** config edited at `/admin/misp`: `enabled`, base `url` (no credentials), `verify_tls`, and event defaults (`default_distribution`/`default_threat_level`/`default_published`). **No secret columns** — the MISP API key is env-only (`ICEBERG_MISP_API_KEY`). Seeded from `ICEBERG_MISP_*` by `services/misp_settings.py`. See *Light-touch IOCs (MISP push)*.
- **WebhookSettings** — the single-row (id=1) **publication-webhook** config edited at `/admin/webhook`: `enabled`, `url` (no credentials), `timeout`, and a non-secret `format` selector. **No secret columns** — the bearer token is env-only (`ICEBERG_WEBHOOK_TOKEN`). Generic JSON is the compatibility default; Slack Block Kit and Teams MessageCard wrappers are opt-in and remain metadata-only. The row is seeded from `ICEBERG_WEBHOOK_*` by `services/webhook_settings.py` (`enabled` seeds true when `ICEBERG_WEBHOOK_URL` was set, so env-only deployments keep firing). Drives the publication webhook (metadata-only POST on publish): `dissemination.enqueue_notifications` snapshots the webhook + proxy rows (non-secret fields only) into a durable `DISSEMINATION_WEBHOOK` `OutboxJob` payload, delivered by the jobs worker after the publish commit — and only for reports disseminable under the TLP ceiling, so a restricted report's title/URL never reaches the endpoint (#203). Modelled on `MISPSettings`/`ProxySettings`. See *Dissemination*.
- **ACHModel** — an Analysis of Competing Hypotheses (Heuer) matrix held against a notebook (roadmap §1c): a key intelligence `question` adjudicated by scoring a matrix of `hypotheses` × `evidence` (both JSON lists of `{id, text}` rows carrying **stable string ids** `h1`/`e1`…, with the `ratings` JSON dict keyed `"{hid}:{eid}"` so deleting a row never re-keys the matrix). Each cell is an `ACHCellRating` (`STRONGLY_CONSISTENT` ++ / `CONSISTENT` + / `NEUTRAL` · / `INCONSISTENT` − / `STRONGLY_INCONSISTENT` −− / `NOT_APPLICABLE`); `NEUTRAL` is the unrated default and is not persisted. The analytic payload is the per-hypothesis **inconsistency score** (`−−`=2, `−`=1, all else 0) — *least inconsistent = most tenable*, and the leading column is flagged. Like DiamondModel/Figure there is **no citation link table**: rendered server-side to a **dynamically-sized SVG matrix** (`render_ach_svg`, all dynamic text XML-escaped) and **embedded inline in a report** via the `[[ach:ID]]` token, resolved (notebook-scoped) at render time for the web view (injected post-nh3), the live preview, and the Typst PDF. CRUD `normalise()`s its matrix (allocate ids, drop blank rows, prune orphan/invalid ratings) so the stored matrix is always consistent. The collection is writer-only; edited on a dedicated Alpine-driven grid (`templates/ach_edit.html`) with a live SVG preview behind `POST /api/preview/ach`. See `services/ach.py`.
- **Report** (intelligence product) — markdown body, `intel_level` (STRATEGIC/TACTICAL/OPERATIONAL), `tlp`, lifecycle `status`, author/reviewer. Also carries optional **ICD 203 structured-judgement scaffolding** — `key_judgements` (the BLUF), `key_assumptions`, `intelligence_gaps` (all markdown) — rendered as discrete sections in the web view and PDF, plus an optional **`analytic_confidence`** (`AnalyticConfidence` LOW/MODERATE/HIGH; nullable = "not stated") — ICD 203 estimative language: confidence *in the judgements*, stamped as a calm neutral marking on the masthead beside TLP/status (web view `confidence_badge` macro + `product.typ`, all formats). *Likelihood* of an assessed event is expressed in prose via the **probability yardstick** (a controlled lexicon → percentage bands, single-sourced in `help_content.py` as `PROBABILITY_YARDSTICK`, surfaced as a collapsible reference panel in the editor and the `estimative-language` glossary entry); the live preview also returns **tradecraft lint warnings** for vague hedging so analysts can tighten language without blocking saves. The web report view and the editor's live/read-only preview share one assembler (`product_html.render_report_product_html` / `preview_report_product_html`, the latter behind `POST /api/preview/product`) so the finished product (Key Judgements callout + body-with-diagrams-and-figures + Assumptions + Gaps) never drifts between them. Cites a subset of its notebook's sources (`ReportSource`) and attachments (`ReportAttachment`), embeds notebook Diamond Models, figures and ACH matrices inline via `[[diamond:ID]]` / `[[figure:ID]]` / `[[ach:ID]]` tokens, embeds its **own ATT&CK technique-coverage matrix** inline via a bare `[[attack]]` token (no ID — the techniques derive from the report's own tags), is classified with taxonomy tags (`ReportTag`), may carry audience-group restrictions, and records optional AI provenance for analyst-accepted AI suggestions. The row carries an ORM **optimistic-locking `version` column** (`version_id_col` — SQLAlchemy bumps it on every persisted update and a stale write raises, mapped to a **409** by `main.py`'s `StaleDataError` handler, so an edit loaded before another request published can't overwrite the finished product) and a `publication_snapshot_hash` naming the report's active **PublicationSnapshot** (empty for drafts).
- **RenderedProduct** — an on-demand PDF for a report (FULL / EXEC_BRIEF / ONE_PAGER). **FULL** renders the masthead, Key Judgements, body, then Key Assumptions + Intelligence Gaps, then the sources/attachments appendix; the **EXEC_BRIEF / ONE_PAGER** are *Key-Judgements-only* products (masthead + markings + judgements, body and caveats omitted). `services/reports.prune_rendered_products` applies configurable retention (`ICEBERG_RENDER_RETENTION_KEEP` / `ICEBERG_RENDER_RETENTION_DAYS`) after each render and via `iceberg-prune-renders`. Each row records the `snapshot_hash` it was built from: a stakeholder may only obtain a PDF built from the report's active publication snapshot, while writer draft renders deliberately keep an empty hash.
- **PublicationSnapshot** — the **immutable publication-time representation** of a finished report (`services/publication.py`): one row per report (unique `report_id`) whose JSON `payload` captures the rendered product HTML, the Typst input, self-contained embedded artefacts (diamond/ACH/attack SVGs, figure bytes) and the MISP push inputs at the moment of publication, keyed by a content `snapshot_hash`. Drafts resolve notebook material live; **the moment a report is published its snapshot is what stakeholders and external egress consume** (`published_html`, `render_snapshot`, `misp_inputs`), so later collection edits/deletes can never rewrite a finished product. `publication.publish` is the **atomic publish path**: it validates APPROVED + reviewer/admin, stamps status/`published_at`, creates the snapshot, writes the synchronous feed `DisseminationEvent`s, the audit row and the durable notification jobs **in one transaction**, maps a concurrent change to a 409 (`PublicationConflict`), and only after the commit kicks the jobs worker. Legacy published reports are frozen by a boot-time `backfill_snapshots`.
- **Requirement** — stakeholder intelligence requirement feeding the analyst tasking board, with a `kind` (`RequirementKind`: **PIR** priority/decision-tied/time-bound, **GIR** standing baseline, **RFI** ad-hoc — default RFI), `priority` + `status`, and PIR-only time-bound scaffolding (`decision_context` free-text + `review_by` date, blanked for GIR/RFI on every write). Traced to the reports/notebooks that satisfy/address it via `ReportRequirement` / `NotebookRequirement`. The **board ordering** blends urgency and kind via `board_rank` (a PIR is floored to at least HIGH so it leads standing/ad-hoc work, but a true CRITICAL item still tops its column) with `kind_rank` as the within-tier tiebreak — see *Requirements & analyst tasking*.
- **Tag** — a term in the controlled CTI taxonomy (`TagKind`: ACTOR/CAMPAIGN/MALWARE/TECHNIQUE/SECTOR/TOPIC; ATT&CK techniques carry their T-code in `external_id`). **Admin-curated** — analysts only *select* tags when classifying a report, never create them; `active=False` retires a tag (kept on historical reports, no longer offered). The "named-threat" kinds (ACTOR/MALWARE/CAMPAIGN — `tags.ALIASABLE_KINDS`) carry a structured **`aliases`** list (a JSON column on `Tag`) so the APT28 / Fancy Bear / Sofacy naming problem resolves to one entity; aliases drive **alias-aware search** (see *Tagging & search*) and render as "Also known as". The same kinds also carry **structured attribution** (roadmap 2b): `suspected_attribution` (free-text sponsor/country), `motivations` (a JSON list validated against the `Motivation` enum — ESPIONAGE/FINANCIAL/HACKTIVISM/DESTRUCTIVE/INFLUENCE, multi-valued via `tags.normalise_motivations`), and free-text fuzzy `first_seen`/`last_seen`. For these kinds `/tags/{id}` renders a dedicated **entity profile** (`templates/entity_profile.html`) — attribution panel + motivation chips + aliases + an ATT&CK link off `external_id` (G-/S-code) + the reports-tagged list; non-named kinds keep the plain `search.html` drill-down. Attribution is admin-edited in `/admin/tags` (gated to named-threat kinds, like aliases) and seeded from `data/starter_tags.json`. See `services/tags.py`.
- **DisseminationEvent** — a published report delivered to a stakeholder's feed, with read tracking.
- **OutboxJob** — a durable, **lease-based unit of external work** (`services/jobs.py`): a `kind` (`JobKind`: `DISSEMINATION_EMAIL` / `DISSEMINATION_WEBHOOK` / `RSS_POLL`), a JSON `payload` (identifiers + non-secret config snapshots only — tokens/passwords stay env-only and are resolved by the worker), a unique `idempotency_key`, attempt/retry counters with exponential backoff (`ICEBERG_JOBS_RETRY_BASE_SECONDS`, capped), and an expiring lease (`lease_token`/`lease_expires_at`, `ICEBERG_JOBS_LEASE_SECONDS`) so a crashed worker can never strand a `RUNNING` job. Routes **enqueue a job in the same transaction as the state that caused it** (the publish transaction, the RSS poll tick); feed `DisseminationEvent`s are deliberately *not* jobs — they stay synchronous. Jobs are processed by an opportunistic post-commit FastAPI background kick (`jobs.schedule_worker` → `process_due_jobs`) and, durably, by the **`iceberg-worker`** console script (one bounded pass by default, `--forever` under a process manager, `--inspect` to list job state); retriable failures return to `PENDING` with backoff, terminal ones stay inspectable as `FAILED` with a truncated `last_error`.
- **AudienceGroup** — admin-managed need-to-know groups. Stakeholders belong to groups via `UserAudienceGroup`; reports are scoped via `ReportAudienceGroup`. A scoped published report is visible/disseminated only to stakeholder members of at least one matching group; unscoped reports keep the previous published-to-all-stakeholders behaviour.
- **ReportEmbedding** — rebuildable local vector row for published reports, used by the related-products API. The default embedding is deterministic and local (`local:hash-v1`) so there is no content egress.
- **ProductFeedback** — the **intelligence-cycle feedback loop** (roadmap backlog D): a stakeholder's feedback on a product **disseminated to them** — an ordinal `usefulness` (`ProductUsefulness` NOT_USEFUL/USEFUL/HIGHLY_USEFUL), an optional `requirement_id` link to **one of the stakeholder's own requirements** the report addresses, a `RfiSatisfaction` verdict (MET/PARTIALLY_MET/NOT_MET) when so linked, and a free-text `comment`. One row per `(report, stakeholder)` (unique constraint; the submit path **upserts**). Feedback is **only** accepted on a delivered product (a `DisseminationEvent` must exist — `was_delivered`), and a stakeholder may only cite a requirement they own *and* the report satisfies. **Closes the loop:** a `MET` verdict from the owning stakeholder auto-advances that requirement to `SATISFIED` (reusing `requirements.set_status`); other verdicts only signal (analysts retain status control). Surfaces on the report view (a stakeholder "Your feedback" card; a writer-only "Product feedback" panel) and the requirement detail (analyst-facing), and feeds effectiveness metrics into the maturity dashboard. Submitted via `POST /api/reports/{id}/feedback` (stakeholder) / `GET` (writer-only list) or the portal `POST /reports/{id}/feedback`. See `services/feedback.py`.
- **Feed** / **FeedItem** — the **inbound collection** channel (FR #50, roadmap backlog I). A `Feed` is an **admin-configured** external RSS/Atom source (`url` unique, `enabled`, `last_fetched_at`/`last_status`/`fetch_error`); a `FeedItem` is a fetched article (deduped per feed on a `(feed_id, guid)` unique constraint, with **nh3-sanitised** `summary`/`content`, `published_at`, and `ingested_at` set when captured). An analyst browses items in the writer-only **feed reader** (`/feeds`) and **"sends to notebook"**, which creates an (auto-graded) `Source` in an existing or new notebook — reusing `services/notebooks.py`. `content` retains the sanitised body as the **seam for future IOC extraction + summarisation** (not built yet). See *Inbound collection (RSS ingestion)*.
- **AuditEvent** — a persisted **security-relevant event** (the local audit trail / forensic source of truth). Carries the OWASP "when/what/where/who/result" attributes: `occurred_at`, `action` (an `AuditAction` value), `category`/`severity`/`outcome`, actor identity (`actor_id` FK→user `SET NULL` + denormalised `actor_email`/`actor_role`), `source_ip`/`user_agent`, `request_method`/`request_path`/`status_code`, `resource_type`/`resource_id`, a `correlation_id`, and a curated `detail` JSON (**never** secrets/tokens/PII). Recorded + emitted to the SIEM via `services/audit.py` + `services/siem.py`. See *Security audit logging → SIEM*.
- **AuditSettings** — the single-row (id=1) runtime SIEM-emit config edited at `/admin/audit`: `enabled`, `methods` (JSON list of `stdout`/`syslog`/`http`), `min_severity`, and non-secret routing (`file_path`, `syslog_host`/`port`/`protocol`/`facility`, `http_endpoint`, `http_verify_tls`). **No secret columns** — the HTTP/HEC token is env-only. Seeded from `ICEBERG_AUDIT_*` defaults by `services/audit_settings.py`.
- **ProxySettings** — the single-row (id=1) **global outbound-proxy** config edited at `/admin/proxy`: `mode` (`ProxyMode` SYSTEM/NONE/EXPLICIT), `proxy_url` (no credentials), `no_proxy` (comma-separated domains + CIDR ranges to bypass). **No secret columns** — proxy credentials are env-only (`ICEBERG_PROXY_USERNAME`/`PASSWORD`), injected at call time. Seeded from `ICEBERG_PROXY_*` by `services/proxy_settings.py`. See *Outbound proxy connectivity*. *(All the single-row settings models — `AuditSettings`/`ProxySettings`/`MISPSettings`/`WebhookSettings` — are lazily seeded through `services/singleton.get_or_create`, an atomic `INSERT … ON CONFLICT DO NOTHING`, so two workers racing their first request can't double-insert the row.)*

## Subsystems & designs

### Report lifecycle (`src/iceberg/services/lifecycle.py`)
`DRAFT → IN_REVIEW → APPROVED → PUBLISHED`, with send-back paths for rework. The author submits their own draft; a `REVIEWER`/`ADMIN` approves, sends back, or publishes. Publishing goes through `services/publication.publish` — one atomic transaction that stamps `published_at`, freezes the **PublicationSnapshot**, writes the synchronous feed events + audit row and enqueues the durable notification jobs, with concurrent changes rejected as 409 (ORM row versioning / `PublicationConflict`). Published reports are immutable — enforced both by the editable-status guards and by the snapshot (stakeholder-facing HTML/PDF/MISP output comes from the frozen snapshot, not live notebook material).

### Requirements & analyst tasking (`src/iceberg/api/requirements.py`, `services/requirements.py`)
Stakeholders (read-only for intel) **submit requirements** — choosing a **kind** (PIR/GIR/RFI; the intake/edit forms reveal the PIR-only decision-context + review-by inputs via Alpine `x-show`); they see only their own. Analysts/reviewers/admins see the **aggregated tasking board** (`/requirements`), grouped by status, and drive each requirement's `status` (OPEN/IN_PROGRESS/SATISFIED/CLOSED). **Board ordering** (the one sort key, shared by the board and the link-picker `_open_requirements`) is `(-board_rank, -kind_rank, created_at)`: a PIR is floored to at least HIGH priority so it always leads standing/ad-hoc work, but a genuine CRITICAL item of any kind still tops its column — so urgent ad-hoc work is never buried by the PIR designation. The board also carries a **PIR coverage panel** (`services/requirements.pir_coverage`, writers only) listing **active** (OPEN/IN_PROGRESS) PIRs that are *collection gaps* (no linked report **and** no linked notebook) or *overdue* (past `review_by`) — overdue is a panel flag only, never fed into the sort (keeps card order deterministic). Analysts establish **traceability** by ticking the requirements a report satisfies (in the report editor) or a notebook addresses (on the notebook page); the requirement detail page shows the linked reports/notebooks. Role split: only stakeholders/admins create requirements; only analysts/reviewers/admins change status and create links; only the owning stakeholder (or admin) edits/deletes a requirement's fields. *(The JSON API `GET /api/requirements` deliberately keeps its `created_at desc` order — board presentation order is a portal concern.)*


### Source reliability grading (`src/iceberg/services/source_grading.py`)
Notebook sources are graded with the Admiralty/NATO system and shown as compact
chips (e.g. `B2`, `B6`) on notebook rows, report citation pickers, report source
lists, and the FULL PDF appendix. Grading is a **fully offline, synchronous local
heuristic** (`heuristic:v1`): reliability is inferred from the source identity
(recognised publisher domain — official / vendor / academic / news / social — or a
named authority in the text), and credibility from the analyst's summary text;
without readable claim content credibility stays `6` ("cannot be judged"). Auto
grading is applied inline on create — there is no outbound network fetch and no LLM
provider integration (these were removed: a corporate deployment shouldn't carry a
server-side URL fetcher / external-API attack surface for what is only a *suggested*
grade). Analysts can manually override, clear, or regrade any source.

### Tagging & search (`src/iceberg/services/tags.py`, `services/search.py`)
A **controlled tag taxonomy** classifies reports by threat actor / campaign / malware / ATT&CK technique / sector / topic. The vocabulary is **admin-curated** (only `ADMIN` creates/edits/retires tags, at `/admin/tags`); analysts *select* from it in the report editor. Tags are classification metadata and are **deliberately editable after a report is published** (CTI re-tags retrospectively) — the tag endpoints guard on `ensure_author` only, not `ensure_editable`. Tags surface as kind-tinted chips in the portal (report view / list / search) and are stamped onto the **rendered PDF** below the masthead (`typst/product.typ` `tag-chip`).

The **starter taxonomy** ships as data (`src/iceberg/data/starter_tags.json`, ~94 entries: CISA sectors, cross-cutting topics, a curated ATT&CK Enterprise technique set, and example threat actors + malware with their ATT&CK G-/S-ids). It is loaded by `load_starter_tags()` and imported idempotently (matched on `(kind, slug)`) by `seed_default_taxonomy()` — run automatically on first boot via `init_db`, and re-runnable as an explicit **import step**: `python -m iceberg.seed` (or the `iceberg-seed` console script), with `--file` (custom taxonomy), `--update` (refresh metadata on existing tags), and `--list` (dry-run summary). CAMPAIGN tags start empty (org-specific events for admins to add).

**Search** (`/search`, `GET /api/search`) is full-text over report title + body + the ICD 203 judgement scaffolding (`key_judgements`/`key_assumptions`/`intelligence_gaps`), plus facets (tag, kind, intel_level, tlp, status), and is **paginated**: `search.search_page` returns a `SearchPage` (results + `total`/`limit`/`offset`), the API takes `limit` (1–100, default 50) + `offset` query params and echoes the totals, and the portal renders pager controls (`search_reports` remains a thin results-only wrapper). The full-text backend is **chosen by SQL dialect** in `search_reports` (`_fts_ids` dispatch): on **SQLite** it's **FTS5** (bm25-ranked) — an external-content `report_fts` table whose indexed columns mirror `report` column names (so the `rebuild` backfills them), created with its sync triggers by an `after_create` event on the `report` table (builds for both `init_db()` and the in-memory test engine); on **PostgreSQL** it's a **DB-maintained generated `tsvector` column** (`report.search_vector` + a GIN index, created by the dialect-guarded `postgres_fts` migration over the same text), queried with `websearch_to_tsquery` and ranked by `ts_rank` — always current, so no reindex step. Semantics differ deliberately by backend (FTS5 prefix terms vs Postgres whole-word stemming), so the SQLite-specific relevance tests stay in `test_search.py` and a backend-agnostic `test_postgres_smoke.py` covers the Postgres path. Search is also **alias-aware** (roadmap 2a): a free-text query is *additionally* resolved against named-threat tag labels + `aliases` (`tags.resolve_alias_report_ids`, a Python scan — the vocabulary is small), and reports tagged with a matching entity are **appended after the bm25 body matches**, so "Fancy Bear" finds APT28-tagged reports even when the body never names the alias. The tag text is deliberately *not* denormalised into the FTS index (that would force re-syncing `report_fts` on every tag edit) — alias resolution stays a query-time tag lookup in `search_reports`, so there is **no FTS DDL change**. **Access control:** stakeholders only ever match *published* reports — `search_reports` reapplies `ensure_visible`'s rule so search can't leak unpublished material.

### Dissemination (`src/iceberg/services/dissemination.py`, `services/email.py`)
On publish, Iceberg matches stakeholders and delivers the report to their **feed**: a stakeholder matches when (a) the report is broadcast-eligible under the TLP ceiling — reports at or below `ICEBERG_DISSEMINATION_MAX_TLP` (default AMBER) are disseminated, while RED / AMBER+STRICT are withheld — **and** (b) the report's `intel_level` equals the stakeholder's `preferred_intel_level` (or the stakeholder has set no preference = all levels), **and** (c) if the stakeholder has tag subscriptions, the report shares at least one subscribed tag, **and** (d) if the report is scoped to audience groups, the stakeholder belongs to at least one of them. Feed delivery is recorded synchronously as `DisseminationEvent`s **inside the publish transaction**; an **email notification** per recipient is enqueued as a durable `DISSEMINATION_EMAIL` `OutboxJob` in that same transaction and delivered after the commit — by the opportunistic in-process worker kick, or by `iceberg-worker` (with lease/retry/backoff) if the process dies first. Email uses a pluggable backend (`ICEBERG_EMAIL_BACKEND`): `console` (default — records to an in-memory outbox + logs, for dev/tests; prod logs a startup warning if it's still `console`) or `smtp`. An optional **publication webhook** POSTs report **metadata only** the same way (a durable `DISSEMINATION_WEBHOOK` job — failure-isolated, proxy-aware, and only enqueued for reports disseminable under the TLP ceiling, so restricted titles/URLs never leave, #203), configured on the single `WebhookSettings` row at **`/admin/webhook`** (`enabled`/`url`/`timeout`/`format`, seeded from `ICEBERG_WEBHOOK_*`) with a connectivity **test** (`dissemination.test_webhook_connection`) — generic JSON is the stable default, while Slack Block Kit and Teams MessageCard formats are opt-in; **no secret columns**, the bearer token is env-only (`ICEBERG_WEBHOOK_TOKEN`); audited `WEBHOOK_SETTINGS_UPDATED`/`WEBHOOK_TEST`. Stakeholders set intel-level and tag-subscription preferences at `/preferences` and read their feed at `/feed`; API clients can also manage tag subscriptions.

**Feedback loop (`services/feedback.py`, roadmap backlog D).** A delivered product is the *only* surface a read-only stakeholder can write to: from the report view they leave **ProductFeedback** (usefulness + optional RFI-satisfaction verdict + comment). The loop closes when the owning stakeholder marks a linked requirement **Met** — it auto-advances to `SATISFIED`. See the **ProductFeedback** domain-model bullet; effectiveness rates roll up into `/maturity`.

### Need-to-know / audience groups (`src/iceberg/api/audience.py`, `web/discovery.py`, `services/reports.py`)
Audience groups are admin-managed via both the JSON API and `/admin/audience`. They assign stakeholder users to named groups and optionally scope reports to one or more groups from the report editor's admin-only **Audience** tab. `reports.ensure_visible` is the shared enforcement point: stakeholders still need `PUBLISHED` status, and when a report has audience groups they must belong to at least one matching group. Search, feed reads, report lists and dissemination route through that same rule so scoped products do not leak by alternate listing paths. Writers/admins retain full library visibility.

### STIX/TAXII export + related reports (`src/iceberg/services/stix.py`, `services/taxii.py`, `services/related.py`)
`GET /api/reports/{id}/stix` exports a visible report as a STIX 2.1 bundle (`application/stix+json`) without turning Iceberg into an IOC store: the report becomes a STIX `report` SDO and controlled taxonomy tags become threat-actor, malware, campaign, attack-pattern or sector identity objects. A read-only TAXII-shaped API rooted at `GET /api/taxii2/` serves one `published-reports` collection: collection metadata, manifest entries for served STIX object ids with report-title metadata, and flattened STIX objects from published reports visible to the authenticated user (stakeholders keep audience-group scoping; drafts are excluded for everyone). Manifest/object pulls support pragmatic TAXII query params for incremental clients: `added_after`, `limit`, `next`, `match[type]`, and `match[id]`. The report view exposes the direct STIX download. `GET /api/reports/{id}/related` uses a local, rebuildable `ReportEmbedding` table for access-scoped related products; embeddings are upserted on publish, shown in a report-view side panel when available, and can be rebuilt with `iceberg-rebuild-related`.

### Governed AI assist (`src/iceberg/services/ai.py`, `api/ai.py`)
AI assist is **off by default** (`ICEBERG_AI_BACKEND=none`). The writer-only API offers advisory endpoints for key judgements, source summaries, tag suggestions, Diamond/ACH starts, analytic challenge notes, and **IOC extraction from a source** (`POST /api/ai/extract-iocs`, the `ioc_extract` task — FR #95). Four backends are selectable via `ICEBERG_AI_BACKEND` (validated at config load): `none`, **`openai-compatible`** (a generic `/chat/completions` endpoint), **`claude`** (Anthropic's first-party API via the official `anthropic` SDK — optional `anthropic` extra; default model `claude-opus-4-8`; key env-only `ICEBERG_AI_API_KEY`), and **`bedrock`** (Amazon Bedrock via the SDK's `AnthropicBedrockMantle` client — optional `bedrock` extra; default model `anthropic.claude-opus-4-8`; `ICEBERG_AI_AWS_REGION` + the standard AWS credential chain, no API key). Backends are **`AIBackend` ABC adapters** in a `_BACKENDS` registry (FR #123): the base `run()` is the template method that owns the cross-cutting plumbing (fail-soft wrapping, the `BackendUnavailable`→`disabled` mapping, the `AISuggestion`+provenance envelope), and each subclass implements only `_complete()` (the provider call + response→dict map). `ClaudeBackend`/`BedrockBackend` extend a shared `_AnthropicBackend` (they differ only in client construction + default model) and deliberately omit `temperature`/`thinking` (Opus 4.x rejects `temperature`); the SDK's `anthropic` import is **lazy** inside `_client()` (a backend enabled without its extra installed fails soft). Adding a provider is a subclass + one registry entry (keep `config._AI_BACKENDS` in sync — config can't import the registry without a layering cycle). All calls pass through `services/ai.py` regardless of backend: report content is TLP-gated by `ICEBERG_AI_MAX_TLP`, failures are fail-soft, prompts/responses are not logged, `AuditAction.AI_ASSIST` records metadata only, and every backend is **proxy-aware** (the global outbound proxy is threaded into the SDK's custom httpx client). The **Diamond/ACH** tasks build their payload from *all* of a notebook's reports, so the gate is applied **per included report** (`ai.sendable_reports` filters every over-ceiling report out of the payload, not just gating on the first — #97); if no report clears the ceiling the task fails closed with no egress. Accepting a suggestion is explicit and stamps `ai_provenance` on the report/source field. **IOC extraction** reads a notebook `Source`'s text (no server-side fetch) and the suggested candidates are refanged + constrained to the curated `IOCType` set by `iocs.normalise_candidates` before the analyst promotes a subset (each accepted row reuses the ordinary `POST /api/notebooks/{id}/iocs` create with the originating source as provenance). Source-content egress (`summarise-source` + `ioc_extract`) is **TLP-gated by the source's own `tlp`** against `ICEBERG_AI_MAX_TLP` (`ai.should_send_source`, mirroring `should_send_report`): an over-ceiling source returns a fail-soft "Source TLP exceeds the configured AI egress ceiling" with no egress. The notebook Indicators section shows the review UI when AI is enabled; the feed-reader surface is a fast-follow (a feed item becomes a `Source` on *send to notebook*, then uses this same flow).

### Inbound collection (RSS ingestion) (`src/iceberg/services/feeds.py`, `web/admin_feeds.py`, `web/feeds.py`)
The **inbound collection** channel (FR #50, roadmap backlog I): external RSS/Atom feeds, **admin-configured** at `/admin/feeds` (CRUD + "Fetch all now"), whose articles populate a **writer-only analyst feed reader** at `/feeds` where an analyst **"sends an article to a notebook"** (existing or new) — creating an auto-graded `Source` via `services/notebooks.py` (`add_source`/`create_notebook`) and stamping `ingested_at`. **One service** (`services/feeds.py`) owns admin CRUD, fetching and ingestion. `fetch_feed` does **bounded, per-feed-isolated outbound HTTP** (`httpx.stream` with `ICEBERG_RSS_FETCH_TIMEOUT` and `ICEBERG_RSS_MAX_RESPONSE_BYTES`, manual redirect handling with every hop re-validated and redirect bodies left unread), parses with **`feedparser`** only after the byte cap is enforced (tolerating `parsed.bozo`), **nh3-sanitises** each item's summary/content (the same boundary as `rendering/markdown.py`), and upserts items deduped on `(feed_id, guid)`; **every failure is logged + recorded on the feed (`fetch_error`), never raised** (mirrors the dissemination per-recipient and `siem._safe` patterns). Fetching is driven through the **durable job outbox**: both the **opt-in scheduled poller** — a lightweight `asyncio` loop in `main.py`'s `lifespan` (gated by `ICEBERG_RSS_POLL_ENABLED`, interval `ICEBERG_RSS_POLL_INTERVAL_MINUTES`, **off by default** so tests/dev never reach the network) — and the admin "Fetch all now" button **enqueue an `RSS_POLL` `OutboxJob`** (committed before any network I/O, so a crash leaves an inspectable row for `iceberg-worker`) and then opportunistically process due jobs in a worker thread (`anyio.to_thread.run_sync` with its own `Session`). Per-feed item counts are capped by `ICEBERG_RSS_MAX_ITEMS_PER_FEED`, and the poller's job idempotency key serialises concurrent ticks so two app processes can't double-fetch the same cycle.

**Security deviation, deliberately contained.** Source-grading intentionally carries **no** server-side URL fetcher (SSRF / external-API attack surface — see *Source reliability grading*). RSS ingestion reintroduces a fetcher but constrains it the same way the SIEM HTTP sink is: **feed URLs are admin-configured only** (analysts never supply one → no SSRF vector from the read path), the poller is **opt-in**, fetches are **timeout- and byte-bounded** and **failure-isolated**, fetched HTML is **sanitised**, and `feeds._validate_feed_url` plus fetch-time target validation provide a defence-in-depth **SSRF guard**: http(s)-only URLs, literal-IP checks on create/update, DNS resolution checks before each fetch, and redirect-hop re-validation (rejecting loopback/private/link-local/reserved/multicast targets; `ICEBERG_RSS_ALLOW_PRIVATE_HOSTS` remains the trusted-internal-feed escape hatch). **No AI yet:** `FeedItem.content` retains the sanitised body as the **seam for future IOC extraction + summarisation** (a future `services/feeds_enrich.py`, gated like the email/SIEM backends). Admin CRUD + fetch are audited (`FEED_CREATED`/`FEED_UPDATED`/`FEED_DELETED`/`FEED_FETCHED`). Rail links: `/admin/feeds` (admin) + `/feeds` (writers).

**Inbound TAXII / MISP pull into notebooks (`services/inbound.py`).** A second inbound staging channel, writer-triggered from a notebook: `POST /api/notebooks/{id}/imports/taxii` fetches a public TAXII/STIX JSON URL through the same bounded, SSRF-guarded fetch path as RSS (`feeds.fetch_bounded_public_payload`) and stages each STIX object as a notebook `Source` (title from the object, raw JSON fenced in `content_md`, TLP AMBER, deduped on a `taxii:{url}#{id}:{modified}` reference); `POST /api/notebooks/{id}/imports/misp` pulls one MISP event by UUID from the **configured** MISP instance (reuses `MISPSettings` + the env-only API key + proxy resolution) and stages it as one `Source` plus one `IOC` per attribute whose type maps into the curated `IOCType` set (deduped against the notebook's existing indicators; unmappable attributes are counted as skipped). Both return created/skipped counts and stay inside the existing trust boundaries: the TAXII URL is writer-supplied but rides the full SSRF guard (URL validation, private-host rejection, byte/timeout bounds, proxy-aware), the MISP pull only ever talks to the admin-configured instance, and everything lands as ordinary reviewable collection material.

### Outbound proxy connectivity (`src/iceberg/services/proxy.py`, `services/proxy_settings.py`, `web/admin_proxy.py`)
A **global** connectivity option for all **outbound HTTP** — for corporate deployments where egress is only allowed via a proxy. **Every outbound HTTP call must honour it when configured**: the RSS fetcher (`services/feeds.py`), the SIEM HTTP sink (`services/siem.py`), the MISP push (`services/misp.py`), the governed AI backend (`services/ai.py`), and the publication webhook (`services/dissemination.py`) all resolve it. **This is a standing rule for any new outbound call** — thread `proxy_settings` to the helper and resolve it; never call `httpx` directly to an external host without going through `proxy.resolve`. (`proxy_settings=None` keeps the previous direct behaviour, so it's a safe default for an unwired path, but new code should wire it.) One admin-managed `ProxySettings` row (edited at `/admin/proxy`) drives three **modes** resolved by the pure helper `proxy.resolve(settings, url) -> httpx kwargs`: **SYSTEM** (default — `trust_env=True`, httpx honours `HTTP(S)_PROXY`/`ALL_PROXY`/`NO_PROXY` env vars; the "honour the system proxy" path and current implicit behaviour), **NONE** (`trust_env=False`, always direct), and **EXPLICIT** (`trust_env=False`, route through `proxy_url` **unless** the target host matches the **no-proxy exclusion list** — standard NO_PROXY semantics: `*`, domain+subdomain suffix, CIDR range, exact IP/`localhost` — in which case go direct). httpx's top-level `get`/`post` take a single `proxy` (no per-host `mounts`), so the bypass decision is made per-URL in `proxy._should_bypass`. **Proxy credentials are a secret** — env-only (`ICEBERG_PROXY_USERNAME`/`PASSWORD`), injected into the proxy URL's userinfo at call time by `proxy._with_credentials`, never persisted on the DB row or shown in the admin page (same discipline as the SIEM HTTP token). In-request callers (RSS fetch, MISP push, AI assist) resolve the proxy with their live session; deferred callers stay **off the request session** by snapshotting the `ProxySettings` row up front — `audit.schedule_emit` snapshots it (alongside the audit-settings snapshot, `model_copy()` → a detached row safe to read after the session closes) and threads it through `siem.emit(...)`, and the publication-webhook path serialises a non-secret proxy snapshot into the durable `DISSEMINATION_WEBHOOK` job payload (`dissemination.enqueue_notifications`) so the jobs worker can honour it without re-reading admin state mid-flight. Admin saves + the connectivity test are audited (`PROXY_SETTINGS_UPDATED`/`PROXY_TEST`); rail link `/admin/proxy` is admin-only.

### Light-touch IOCs (MISP push) (`src/iceberg/services/iocs.py`, `services/misp.py`, `services/misp_settings.py`, `web/admin_misp.py`)
A **light-touch indicator staging** layer — Iceberg is **not** an IOC store; the authoritative store stays external (**MISP** first). Three moves: (1) an analyst records **IOCs as notebook entities** (`IOC`, writer-only CRUD via `services/iocs.py` mirroring `services/diamond.py` — notebook-scoped `get_scoped`, optional same-notebook `source_id` provenance; entered **manually** or **AI-suggested** from a source via the governed `ioc_extract` task — `iocs.normalise_candidates` refangs `hxxp`/`[.]` forms and drops anything outside the `IOCType` set, and the analyst promotes accepted candidates through the same create path, FR #95); (2) a report **cites a subset** (`ReportIOC` link table, `reports.set_ioc_citations` — own-notebook only, `ensure_editable` so it's publish-immutable), rendered as a structured **Indicators appendix** (type · value · context) in the web report view (`report_view.html`) and the **FULL** Typst PDF (`product.typ` appendix, alongside sources/attachments); (3) a writer **pushes the cited indicators to MISP as one event** via `services/misp.push_report` — `IOCType.value` *is* the MISP attribute type (1:1 map, `_CATEGORY` supplies each type's MISP category), the report TLP + report tags become MISP event tags, and **each indicator's own `tlp` rides as a per-attribute tag** so MISP honours per-indicator markings. **TLP egress gate (push-everything-with-confirmation):** cited IOCs above `ICEBERG_MISP_MAX_TLP` (default AMBER) are **not filtered or blocked** — instead `push_report(..., acknowledge_tlp=False)` returns a `needs_confirmation` outcome (no POST) listing the over-ceiling markings, and the report-view push card prompts the writer (`confirmSubmit` + a hidden `acknowledge_tlp`) before re-pushing; `over_ceiling_iocs` computes the set for both the gate and the warning banner. The push is **lifecycle-gated** (`misp.can_push_report` — only **APPROVED or PUBLISHED** reports may egress; a draft/in-review push records an error outcome with no POST, #180), **idempotent** (the `ReportMispEvent` row stores the `event_uuid`; first push `POST /events/add`, subsequent pushes `POST /events/edit/{uuid}`), **serialised against concurrent pushes** (`misp._reserve_push` takes a lease-token reservation on the `ReportMispEvent` row before egress, so two writers clicking Push race safely), and **failure-isolated** (config/transport errors are recorded on the row's `last_status`/`error`, **never raised** — same contract as `siem.emit`), surfaced on the report view's "Push to MISP" card. A **published** report's push reads its indicator set from the frozen `PublicationSnapshot` (`publication.misp_inputs`), so post-publication notebook edits can't alter what leaves the org. Reuses the SIEM/proxy discipline: bounded `ICEBERG_MISP_TIMEOUT`, the global outbound proxy (`proxy.resolve`), admin `verify_tls`, and the **API key env-only** (`ICEBERG_MISP_API_KEY`, sent as the `Authorization` header at call time, never on the DB). Admin config is the single `MISPSettings` row at **`/admin/misp`** (`services/misp_settings.py`, seeded from `ICEBERG_MISP_*`) with a connectivity **test** (`misp.test_connection`). Triggered manually (not auto-on-publish). Audited: `IOC_CREATED`/`IOC_UPDATED`/`IOC_DELETED`, `MISP_SETTINGS_UPDATED`/`MISP_TEST`, `MISP_PUSHED` (WARNING). Rail links: `/admin/misp` (admin), the report-view push card + editor Citations-tab indicator picker (writers). New `IOC`/`ReportIOC`/`ReportMispEvent`/`MISPSettings` tables (one migration). **No inline `[[ioc:ID]]` token** — IOCs are a cited appendix, not a body embed.

### Security audit logging → SIEM (`src/iceberg/services/audit.py`, `services/siem.py`, `auth/audit_middleware.py`)
Security-relevant events are **persisted locally** (the `AuditEvent` table — the forensic source of truth, surviving a SIEM outage) **and emitted as structured JSON** to a SIEM following the full **OWASP Application Logging** event-attribute set via `audit.to_owasp_dict`: **when** (event + log datetime in international/UTC format + interaction id), **where** (application name/version/host, service, entry-point URL + HTTP method), **who** (source IP + user identity), **what** (type, severity, a `security_relevant` flag, a human-readable `description`, result/outcome, resource, curated `detail`). The `description` is auto-derived from the structured fields when a caller doesn't supply one (`audit._describe`), and both it and the source IP surface in the `/admin/audit` trail. `source_ip` is `request.client.host`, so behind a reverse proxy it is accurate **only because the container runs uvicorn with `--proxy-headers`** (trusting `X-Forwarded-For`/`-Proto` per `FORWARDED_ALLOW_IPS`); without it the proxy/ingress IP would be recorded — see the Dockerfile + *TLS / running behind a proxy* in the README. Two capture mechanisms feed one `audit.record` sink: (a) **explicit instrumentation** at semantically rich success events — login/logout (`auth/routes.py`), report lifecycle transitions (`api/reports.py` + `web/reports.py`, `audit.lifecycle_action` maps status→action; publish is elevated to WARNING and carries the dissemination recipient count), admin tag curation (`api/tags.py` + `web/discovery.py`), and sensitive-file access (`api/notebooks.py` attachment/figure upload/download/delete); and (b) a **central `AuditMiddleware`** registered *outermost* in `main.py` so it observes the final response — recording `AUTHZ_DENIED` (401/403 role-guard denials) and `CSRF_BLOCKED` (it replicates the CSRF middleware's same-origin test to classify the 403). Every request gets a `correlation_id` (on `request.state`) tying a middleware-recorded failure to an explicitly-recorded success. **OWASP discipline:** the curated `detail` JSON must never carry secrets/tokens/passwords/JWTs/file bytes. *Note:* an audit `record_and_emit` commits the session, expiring ORM instances — routes that return an ORM object `session.refresh(...)` it afterwards.

**Pluggable SIEM emit (`services/siem.py`, modelled on `services/email.py`).** `emit(event_dict, settings)` dispatches to every enabled method when auditing is on and the event meets the configured min-severity: **`stdout`** (JSON line on the `iceberg.audit` logger + optional file — the baseline a sidecar shipper forwards; inspectable test `OUTBOX`), **`syslog`** (RFC 5424 over UDP/TCP), and **`http`** (JSON POST to a Splunk HEC / Elastic / webhook endpoint via `httpx`, bearer token from env). Each method is isolated — a failing/unreachable sink is logged but never raises, and emit runs off the response path (a background task / `response.background`).

**Application logging (`logging_config.py`).** Ordinary `iceberg.*` app logs are configured once at app startup from env: `ICEBERG_LOG_LEVEL` and `ICEBERG_LOG_FORMAT` (`auto` = text outside prod, JSON in prod). `AuditMiddleware` also seeds a contextvar with the request `correlation_id`, so app logs emitted during a request line up with the audit trail; non-request logs use `correlation_id="-"`. The `iceberg.audit` stdout SIEM payload is compatibility-sensitive and passes through as the raw OWASP JSON line rather than being double-encoded into the app-log JSON envelope. Uvicorn's own loggers remain server-owned.

**Admin-configurable (`web/admin_audit.py`, `templates/admin_audit.html`).** Runtime routing config lives on the single `AuditSettings` DB row (enable, chosen methods, min severity, file path, syslog host/port/protocol/facility, http endpoint, verify-TLS), edited live at the admin-only **`/admin/audit`** console (settings form + filterable event trail + "Send test event"). **Secrets stay in env** (`ICEBERG_AUDIT_HTTP_TOKEN`) and are never written to the DB; `services/audit_settings.py` seeds the row from `ICEBERG_AUDIT_*` env defaults on first read. Rail link (`ic-audit`) is admin-only.

### Frontend / design system
The portal is a **"command-center" design** — a light, print-like, authoritative skin over a dense, persistent app shell. Styling lives in a hand-authored stylesheet `src/iceberg/static/css/iceberg.css` (oklch colour tokens, component classes like `.card` / `.btn` / `.tag` / `.board` / `.md`), served from the existing `/static` mount and linked in `base.html`. A compiled **Tailwind** utility build (mapped to the same CSS variables) supplies layout utilities, alongside **Alpine.js**. Three fonts carry meaning: **Archivo** (UI/headings), **JetBrains Mono** (data/IDs/markings), **Spectral** (finished-product prose — the editor preview and published report body, never UI chrome).

**Self-hosted assets + SRI (FR #58, `scripts/vendor_assets.py`).** Tailwind, Alpine and the three fonts are **first-party** — served same-origin from `/static`, never from a CDN — at **pinned exact versions** with **Subresource Integrity**. `scripts/vendor_assets.py` (no Node) is the reproducible regenerator: it downloads Alpine, compiles a minified `static/css/vendor/tailwind.css` via the **standalone Tailwind CLI binary** (**Tailwind v4**, CSS-first config in `frontend/input.css` — `@import "tailwindcss" source(none)` + `@source` for the scanned templates/JS + an `@theme inline` block mapping the colour/font utilities onto the oklch tokens; no JS config file), self-hosts the Google Fonts as `static/fonts/*.woff2` + a generated `static/css/vendor/fonts.css`, computes `sha384` SRI, and writes the `static/assets.lock.json` manifest. **Alpine** is vendored as its **CSP build** (`@alpinejs/csp`, v3) — it evaluates directive expressions with a custom interpreter (no `eval`/`new Function`), which is what lets `script-src` drop `'unsafe-eval'`. `templating.py` exposes that manifest as the Jinja global `assets`, and `base.html` renders the vendored `<link>`/`<script>` with `integrity="{{ assets.*.integrity }}"`. SRI is on the *vendored* assets only — the hand-edited `iceberg.css`/`tags.js` stay hash-free. The vendored subtree is excluded from biome lint (`biome.jsonc`). This removed all CDN origins and, with the CSP build, **unblocked the strict CSP** (issue #57 — now enforced; see *Security response headers*). See *Maintenance* for the version-bump procedure.

**Alpine under the CSP build — no inline JS (`static/js/tags.js`).** The CSP build cannot `eval`, so two rules hold: (1) **every component is registered** via `Alpine.data()` in `static/js/tags.js` (one registry, loaded *before* the Alpine `<script>` so its `alpine:init` listener is in place; `x-data="name(arg)"` names a registered component) — there are **no inline `x-data` object literals with methods**; (2) the build **prohibits `x-html`** (use `x-ref` + `x-effect="renderX()"` writing `innerHTML` in native JS) and **all inline `on*=` handlers / inline `<script>` blocks** (use Alpine `@`-handlers bound to component methods). Server state reaches a component through a **`<script type="application/json">` island** read with `JSON.parse` in the factory (the CSP interpreter does **not** decode `tojson`'s `\uXXXX` escapes, so HTML/SVG/prose passed via an attribute expression would corrupt) — small scalar args (ids, enum values) may still be passed inline. `tests/test_frontend_assets.py` guards these invariants (no inline handlers, no inline `<script>`, no `x-html`).

**App shell (`base.html`).** A fixed-height flex shell: a **persistent role-aware left rail** (240px, deep cool charcoal — `--rail*` tokens) with grouped nav + identity card + logout, and a **workspace** column = a sticky 56px **topbar** (breadcrumb · ⌘K trigger · stakeholder feed bell) over a scrolling **`.canvas`** that wraps the unchanged `{% block content %}` in a `.canvas-inner` (max ~1340px; `page-narrow`/`page-wide`/`canvas--flush` modifiers). The rail nav is server-rendered from `user.role` (groups differ per role) with **active state still derived from `request.url.path`**; a hidden inline SVG **icon sprite** (`#ic-*` symbols) feeds `<use>` references. Collapses to an icon strip under 960px. The iceberg mark is inline SVG in `templates/_glyph.html` (included by `base.html` and `login.html`; login renders with no rail/topbar since `user` is falsy).

**⌘K command palette.** An additive Alpine overlay in `base.html` (jump-to over the role's nav items, built as a Jinja `jump_items` list); ⌘K/Ctrl-K or the topbar trigger opens it, arrow keys + Enter navigate, Esc closes. It follows the combobox/listbox dialog pattern: focus stays in the palette while open, the active result is exposed with `aria-activedescendant`, and focus returns to the invoking control on close. No backend dependency (live `/api/search` results are a noted future hook).

**Report editor (`report_edit.html`).** A full-height 3-pane workspace (`.editor-shell`): a command header (inline title + lifecycle flow + save-state + Submit/Approve/Publish), then a `1fr 1fr 340px` grid of **markdown ∣ live preview ∣ docked tabbed panel** (Citations · Tags · AI review for editable drafts · Reqs · Audience for admins · Render), with intel-level/TLP/confidence pinned to the dock footer. The dock uses standard tab/tablist/tabpanel semantics with arrow/Home/End roving selection; the taxonomy picker is a combobox/listbox with keyboard add, removal and dismissal. The AI review tab requests advisory judgement, controlled-tag and analytic-challenge suggestions only: analysts edit them locally, explicitly apply them through the ordinary report/tag save flows, and accepted report fields receive AI provenance after a successful save. Disabled/TLP/provider messages remain inline and fail-soft. The markdown pane's **Insert ▾** toolbar drops `[[diamond|figure|ach:ID]]` tokens at the cursor. The single product `<form id="reportform">` is wired across header/pane/footer via the HTML `form=` attribute (no nested forms); it keeps the existing per-section POSTs and live-preview `POST /api/preview/product` binding verbatim, plus **debounced autosave** (1.2s) to the same `/reports/{id}` endpoint behind the header save-state indicator.

This is a skin + layout reorganisation over the same routes/forms/Alpine bindings (`tagPicker`/`taxonomyFilter`, the preview/ach/diamond `x-data` blocks, citation autosave) — no behavioural coupling. **Two additive read-only index routes** back the rail: `GET /notebooks` (`notebooks_list.html`, writer-only) and `GET /tags` (`entities_list.html`, named-threat entity browse).

### Rendering
- Markdown → sanitized HTML (`src/iceberg/rendering/markdown.py`, markdown-it-py + nh3) for the live preview and portal display.
- Report → PDF: **via the Typst binary** (`src/iceberg/rendering/typst.py`). Markdown is rendered inside Typst by the `cmarker` package (fetched from the Typst registry on first use; version pinned in `src/iceberg/typst/product.typ`). If Typst is not installed, render endpoints return 503.

### ATT&CK Navigator export & coverage matrix (`src/iceberg/services/attack.py`)
A **pure derivation** over the existing taxonomy (roadmap backlog A) — **no new model, no migration**. `TECHNIQUE` tags already carry the MITRE ATT&CK technique id in `Tag.external_id` (e.g. `T1566`) and the ATT&CK *tactic* name in `Tag.description` (e.g. "Initial Access" — per `data/starter_tags.json`). From those, `services/attack.py` emits (a) a schema-conformant **ATT&CK Navigator layer** (`.json`, versions/domain pinned) for a single **report** (`report_layer`, techniques scored 1) or a named-threat **entity** (`entity_layer`, techniques scored by occurrence across the entity's reports), served as a download by `api/attack.py` (`GET /api/attack/reports/{id}/layer`, `GET /api/attack/tags/{id}/layer`); and (b) a **technique-coverage matrix** (`coverage_matrix`) grouping technique frequency into ATT&CK tactic columns (`TACTIC_ORDER`, unmatched descriptions bucketed `Uncategorised`) for the in-portal heatmap — a global `/matrix` page (`web/discovery.py` `matrix_view`) plus a per-entity mini-heatmap on `/tags/{id}` (shared `templates/_attack_matrix.html` macro, `.heat-*` shading in `iceberg.css`); and (c) an **inline report embed** — a bare `[[attack]]` token (FR #41) renders the report's *own* technique coverage as a hand-built, dynamically-sized **SVG matrix** (`render_attack_svg`, all dynamic text XML-escaped — same discipline as the Diamond/ACH SVGs; `report_attack_svg(report)` returns the SVG or `None` when the report has no `TECHNIQUE` tags) embedded **inline at the token's position** in the web view, the live preview (injected post-nh3 through `services/product_html.py`'s shared substitution pass, alongside diamond/figure/ach) and the Typst PDF (token rewritten to an `attack.svg` image in the per-render `--root`, reusing `product.typ`'s generic `image` override). The token is bare (no ID) because a report's techniques come from its own tags; the live preview resolves them against the report's *saved* tag set. A report with no technique tags degrades to an "unavailable" notice like the other unresolved tokens. **Access-scoped like search:** report layers go through `reports.ensure_visible` (stakeholder → 404 on an unpublished report) and entity/matrix aggregation rides `search.search_reports` (published-only for stakeholders), so nothing leaks. The entity-layer endpoint is restricted to `tags.ALIASABLE_KINDS`. (The `description`-as-tactic coupling is a soft convention; a dedicated `tactic` column is a noted future hardening.)

**Full ATT&CK technique import (`services/attack_import.py`, `attack_import.py` CLI).** The starter taxonomy's curated technique set can be replaced/refreshed from the **complete MITRE Enterprise ATT&CK catalogue** with the **`iceberg-import-attack`** console script — deliberately **file-only** (an operator obtains and pins the Enterprise ATT&CK STIX bundle through their normal supply-chain process; the server never fetches TAXII/GitHub/arbitrary URLs), with an optional `--sha256` pin verifying the reviewed file and `--update` to refresh metadata on existing tags. The importer parses the bundle, maps techniques + sub-techniques to `TECHNIQUE` tags (revoked/deprecated techniques are **retired** — `active=False` — rather than dropped, matching the taxonomy's retire semantics) (T-code in `external_id`, tactic name in `description` — the same convention the coverage matrix reads), and touches **only** controlled `TECHNIQUE` tags — never sources, IOCs or reports.

### CTI program maturity & effectiveness dashboard (`src/iceberg/services/maturity.py`)
A **pure derivation** over existing data (roadmap backlog H) — **no new model, no migration** — backing a **writer-only `/maturity`** portal page (`web/discovery.py` `maturity_view`, gated by `_require_writer`; no API, like `matrix_view`/`help_view`). `program_maturity(session)` follows the `coverage_matrix`/`pir_coverage` convention (return plain dicts/lists structured for the template) and surfaces four program-health groups computed by direct unscoped queries (writers see all reports): **production** (reports by `status`, 30/90-day publish velocity, median time-to-publish, intel-level mix, reviewer engagement), **requirement coverage** (counts by kind/status, satisfaction rate over non-CLOSED, and an *all-kinds* coverage rate generalising `requirements.pir_coverage`'s "no linked report **and** no notebook" gap test — `pir_coverage` is reused verbatim for the PIR gap/overdue slice), **dissemination reach** (`DisseminationEvent` totals, distinct stakeholders reached, feed read-rate via `read_at`, median delivery lag, the TLP-withheld fraction using the same `is_disseminable` ceiling as `services/dissemination.py`, plus the **feedback-loop** effectiveness rates — response / useful / RFI-satisfaction — from `feedback.feedback_effectiveness`), and **tradecraft adoption** (share of **published** reports applying source grading, ICD-203 judgement scaffolding, `analytic_confidence`, an embedded analytic token `[[ach|diamond|figure:…]]`/`[[attack]]`, and ≥1 `TECHNIQUE` tag). On top sits an **indicative CTI-CMM rollup**: a `_level(value, t1, t2, t3)` helper buckets a 0–1 metric into **CTI0 (Pre-foundational) → CTI1 (Foundational) → CTI2 (Advanced) → CTI3 (Leading)** for four capability dimensions (analytic tradecraft, **stakeholder alignment** — requirement coverage blended with feedback-loop satisfaction once verdicts exist, production discipline, dissemination), with an overall level and a `disclaimer` framing it as evidence for — **not a substitute for** — a formal [CTI-CMM](https://cti-cmm.org/) self-assessment. The page (`templates/maturity.html`) renders a KPI `stat-grid`, the rollup as level-chipped bars, and per-group distribution panels (hand-built CSS bars, house style — no JS chart lib; `.mdim`/`.lvl-*`/`.distbar` in `iceberg.css`). Rail link (`ic-maturity`) is writer-only.

### Diamond Model diagrams (`src/iceberg/services/diamond.py`)
A Diamond Model is rendered to a **self-contained SVG** by hand-built string templating (no extra deps; same spirit as the inline `_glyph.html` mark). One generator (`render_diamond_svg`) feeds three surfaces from a single source: (a) **web report view / live preview** — the `[[diamond:ID]]` token is swapped for an inline `<figure>` **after** nh3 sanitisation (nh3 would otherwise strip server `<svg>`); the figure is safe because every vertex value is XML-escaped at generation; (b) **Typst PDF** — `render_report` writes each referenced diagram to `diamond-{id}.svg` in the per-render temp `--root` and rewrites the token to a markdown image, so it embeds **inline at the token's position**. `product.typ` overrides cmarker's `image` in its `scope` so the path resolves against the template dir (the temp root), not the cmarker package, and constrains the diagram to 92% column width. Tokens are **notebook-scoped**: a token resolves only to a diamond in the report's own notebook; unknown / cross-notebook ids degrade to an "unavailable" notice. The diagram **labels its two meta-axes** (`↕ SOCIO-POLITICAL` adversary↔victim, `↔ TECHNICAL` capability↔infrastructure — the Diamond Model's defining idea), renders confidence as an **ordinal pip-meter** (not a 4th red/amber/green stamp competing with TLP/status), and exposes accessibility via `<title>`/`<desc>` (so the title lives in escaped text content, never a raw attribute).

### ACH matrices (`src/iceberg/services/ach.py`)
An ACH (Analysis of Competing Hypotheses) matrix is rendered to a **self-contained, dynamically-sized SVG** by hand-built string templating (same spirit as `render_diamond_svg`; viewBox width/height derive from the hypothesis-column × evidence-row counts — the key departure from the diamond's fixed canvas). One generator (`render_ach_svg`) feeds three surfaces: (a) **web report view / live preview** — the `[[ach:ID]]` token is swapped for an inline `<figure>` **after** nh3 sanitisation, sharing `services/product_html.py`'s single substitution pass with diamonds + figures; safe because every hypothesis/evidence/question value is XML-escaped at generation; (b) **Typst PDF** — `render_report` writes each referenced matrix to `ach-{id}.svg` in the per-render temp `--root` and rewrites the token to a markdown image (reusing `product.typ`'s generic `image` scope override — no template change). Tokens are **notebook-scoped**; unknown / cross-notebook ids degrade to an "unavailable" notice. Each cell carries a **glyph + colour** (both encode the rating — accessibility, like the diamond confidence pip-meter being ordinal); a **summary row** shows each hypothesis's inconsistency score with the leading (least-inconsistent, most-tenable) column flagged, and `<title>`/`<desc>` rank the hypotheses for screen readers. The matrix is captured per notebook on an **Alpine-driven editable grid** (`templates/ach_edit.html`: add/remove hypothesis columns + evidence rows, a per-cell rating `<select>`, live SVG preview via `POST /api/preview/ach`, submitted as one JSON `matrix` field); the service `normalise()`s it on every write (stable ids, dropped blanks, pruned orphan/invalid ratings). See the **ACHModel** domain-model bullet above.

### Figure embedding (`src/iceberg/services/figures.py`, `services/product_html.py`)
An analyst uploads an image to a notebook's **Figures** collection (PNG/JPEG/GIF — the browser-`data:`-URI ∩ Typst-`image()` intersection; WebP/SVG are excluded) and embeds it inline by writing a `[[figure:ID]]` token in a report's body — mirroring the Diamond Model token, including notebook-scoping and the "unavailable" degrade for unknown/cross-notebook ids. Two surfaces from one upload: (a) **web view / live preview** — the token is swapped for an inline `<figure>` whose `<img>` carries a base64 `data:` URI of the file's bytes, injected **after** nh3 sanitisation (nh3 would otherwise strip a `data:` URI); the caption/alt is HTML-escaped, so it's safe; (b) **Typst PDF** — `render_report` copies each referenced image into the per-render `--root` as `figure-{id}{ext}` and rewrites the token to a markdown image, reusing `product.typ`'s generic `image` scope override (no template change). The shared web/preview pipeline lives in **`services/product_html.py`** (the assembler moved out of `diamond.py` so one `_to_html` pass substitutes *both* diamond and figure tokens). Figures are a **token-only** embed — there is no `ReportAttachment`-style link table and embedded figures do **not** appear in the PDF appendix. The figures collection is writer-only, but the published report inlines the bytes as a `data:` URI, so report viewers (incl. read-only stakeholders) see embedded figures without hitting the writer-only `/raw` endpoint (which serves the notebook/editor management thumbnails only).

### Database & migrations (`src/iceberg/db.py`, `src/iceberg/migrations/`)
The datastore is driven entirely by **`ICEBERG_DATABASE_URL`**: **SQLite** is the
zero-dependency **local dev/test default only**; **PostgreSQL** (`postgresql+psycopg://…`, the
`postgres` extra → `psycopg` v3) is the **required production/container datastore** — prod boot
refuses a SQLite URL (`config._guard_production`). `db.py` branches engine config on the
URL — SQLite keeps `check_same_thread`; networked backends get `pool_pre_ping=True`. The
`_configure_sqlite` connect-event (WAL / `foreign_keys=ON` / `busy_timeout`) is guarded on the
driver connection type so it's a no-op on Postgres (and correct for the tests' own engines,
including a Postgres test engine). The migrate **deploy entrypoint is `iceberg-migrate`**
(`maintenance.migrate_main` → `run_migrations`), which uses the in-code Alembic config (no
packaged `alembic.ini` needed) and is what the k8s migrate Job runs.

Schema is owned by **Alembic** (SQLModel models are the source autogenerate compares
against). `init_db()` runs `alembic upgrade head` on boot when `ICEBERG_AUTO_MIGRATE=true`
(the default — convenient for dev), then seeds the taxonomy and rebuilds the FTS index (both
idempotent); set the flag false in prod and run migrations in the deploy step. Migrations run
against the app's own engine via a **shared connection** (`db.run_migrations` injects it into
`env.py`) so an in-memory `sqlite://` DB works too. **Backend-specific search objects are not in
SQLModel metadata and are dialect-guarded in the migrations**: the baseline migration owns a
frozen copy of the SQLite `report_fts` virtual table + sync triggers (created only when
`dialect.name == "sqlite"`), and the `postgres_fts` migration owns the Postgres `report.search_vector`
generated `tsvector` column + GIN index (created only when `dialect.name == "postgresql"`).
`env.py` excludes `report_fts*` (table) and `search_vector` / `ix_report_search_vector` (column +
index) from autogenerate so it never tries to drop them. `env.py` uses `render_as_batch=True` for
SQLite ALTER support. Tests keep the fast in-memory `create_all` path; `tests/test_migrations.py`
runs a real `upgrade head` on a temp DB and asserts (via `alembic check`) that models and
migrations don't drift, and the CI **`postgres-smoke`** job runs the suite's Postgres path against
a real Postgres service. New change → `alembic revision --autogenerate -m "..."`; pre-existing DBs
built by the old `create_all` → `alembic stamp head` once.

**Multi-replica caveat:** Postgres removes the database single-writer bottleneck, but
attachments/figures/rendered PDFs are still written to a **local disk** path
(`ICEBERG_ATTACHMENTS_DIR`/`FIGURES_DIR`/`RENDER_OUTPUT_DIR`), so horizontal scaling still needs
shared file storage (RWX volume or object store) — a separate follow-on. The k8s Deployment stays
`replicas: 1` + `Recreate` until then.
